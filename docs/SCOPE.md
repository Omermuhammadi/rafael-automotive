# SCOPE.md — What's In, What's Out, and Why

This is the contract behind the build. When a request or the PDF pushes past these lines, this
file is the reason to hold the line.

This file fixes **what** we deliver. **How** it's built — tools, libraries, approach — is the
engineer's call (`CLAUDE.md §4`), and all test data is generated by the build itself
(`CLAUDE.md §5`). Nothing here depends on the client handing over files.

---

## In scope (the accepted offer)

- **Pillar A — Binary Studio**, end to end:
  - Load `.bin` / `.hex` / `.s19`, hex viewer with offsets, CALID/CVN read.
  - Apply byte patches at **defined offsets** from an external patch file.
  - Checksum validate + repair (CRC16 / CRC32 / block-sum).
  - Hardware bridge: UDS read/write-back code (J2534/Tactrix), **software-validated** here.
- **Pillars C + D — OCPP Sniffer**, end to end:
  - Mock CSMS, simulated charge point, colour-coded frame log, fault triage,
    JSON deep-dive inspector, remote station reset.

---

## Out of scope (do not build)

- **Pillar B — live OBD-II stream (Tab 2).** Not in this offer.
- **Automatic map discovery ("hunting matrix").** Using existing ECU definitions / known offset
  tables to locate maps is fine and encouraged; building a from-scratch engine that reverse-
  engineers maps out of *unknown* binaries is not. → *reason below.*
- **Universal manufacturer-checksum engine.** CRC16/32 + block-sum, plus at most one specific
  scheme for the single target ECU. → *reason below.*
- **"Multi-vehicle" support.** One target ECU family for the MVP.
- **On-vehicle validation of the read/write path by us.** We have no cable and no car; the
  client validates the real Tactrix path on their vehicles. → *the hardware boundary.*
- **OCPP 2.0.1.** MVP uses 1.6J only.

## Why these lines exist

**Map discovery is a research problem, not a feature.** Finding maps in an arbitrary binary is
different for every ECU and normally needs the definition files that commercial tools (WinOLS
et al.) charge thousands for. It has no clean "done" state and is the single most likely thing
to blow a 7-day schedule. Even the expensive tools don't conjure maps — they ship definition
databases. So we make the tool the *scalpel* (apply bytes at a known offset) and the human/
definition file the *surgeon* (decide where). The PDF's own wording, "scans fixed offsets," is
the doable version.

**Manufacturer checksums are a rabbit hole.** Plain CRC is a few lines. Every ECU family rolls
its own scheme (sometimes nested), each reverse-engineered separately. Pinning one target ECU
turns this from infinite to finite.

**The hardware boundary is physics, not caution.** The J2534 driver only talks to the real USB
cable; the seed/key security handshake only completes against a real ECU. Neither can be
verified without the hardware. So the UDS logic is proven against a **software mock ECU**, and
the real path plugs into the same `Transport` seam for the client to validate. The client's
targets — **2014 Dodge Charger 3.6** and **2015 Chevy Silverado 1500 5.3** — are modern locked
ECUs whose free open-source seed/key is not a given; that uncertainty is precisely why the seam
exists and why on-vehicle validation sits on the client's side.

## Design/repo note (keep the tool neutral)

The application is a **general-purpose byte patcher**: it applies bytes at offsets from an
external patch-definition file that the *user* supplies. Specific patch definitions are **not**
hardcoded and **not** committed to this repository. This keeps the codebase a neutral
instrument — like a hex editor — with the specific modifications living in the user's own
external config, which is their domain and responsibility. Do not bake specific vehicle
modifications into the repo.
